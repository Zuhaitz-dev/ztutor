# Conditional Compilation: The Linux `CONFIG_SMP` Architecture

## `#ifdef` vs `if`: Compile-Time vs Runtime

This distinction matters enormously for systems code:

```c
// RUNTIME branch: both paths exist in the binary
// The branch instruction is always there; the dead path is just not taken
if (CONFIG_SMP) {
    spin_lock(&lock);
}

// COMPILE-TIME branch: the excluded path DOES NOT EXIST in the binary
// Zero overhead, zero code, zero memory
#ifdef CONFIG_SMP
    spin_lock(&lock);
#endif
```

An `if` at runtime still generates assembly for both branches. A `#ifdef` at compile time eliminates one branch from the binary entirely: not dead code, not a no-op, but *absent*. For an OS kernel that must be maximally efficient on the least capable hardware, this difference is everything.

## How `#ifdef` Works

```c
#ifdef SYMBOL      // if SYMBOL is defined (any value, even empty)
    // included
#else
    // excluded
#endif

#ifndef SYMBOL     // if SYMBOL is NOT defined
    // included
#endif

#if defined(A) && defined(B)   // boolean combinations
    // included when both A and B are defined
#endif
```

Macros are defined either via `#define` in source or via the compiler command line:

```bash
gcc -DCONFIG_SMP -DDEBUG_LEVEL=2 file.c
```

This is how build systems (Make, CMake, Meson) inject configuration into source files.

## The Linux Kernel's Scale

The Linux kernel runs on hardware ranging from a Raspberry Pi Zero (1 CPU, 512 MB) to IBM Z mainframes (hundreds of cores, terabytes of memory), all from the same source tree. The `make menuconfig` (or `make defconfig`) step generates `include/generated/autoconf.h`:

```c
// Generated by Kconfig: do not edit manually
#define CONFIG_SMP 1
#define CONFIG_HZ 250
#define CONFIG_X86_64 1
#define CONFIG_PREEMPT_VOLUNTARY 1
#define CONFIG_MODULES 1
// ... thousands more
```

Every source file includes this header, directly or transitively. Preprocessor conditionals throughout the codebase use these macros to include or exclude entire subsystems. `CONFIG_SMP` controls multi-processor locking, `CONFIG_IPV6` controls the IPv6 network stack, `CONFIG_BLK_DEV_NVME` controls the NVMe storage driver, and `CONFIG_DEBUG_KERNEL` gates debug-only code paths. A minimal embedded kernel might be 1 MB; a full-featured server kernel might be 15 MB. Most of the difference is the result of conditional compilation.

## Zero-Overhead Abstraction: The UP Spinlock

Spinlocks protect shared data structures in SMP (multi-processor) systems, where two CPU cores cannot safely modify the same data simultaneously. On a uniprocessor (UP) system, however, there is no parallelism, and spinlocks represent wasted cycles.

Linux routes the choice at the header level:

```c
// include/linux/spinlock.h
#ifdef CONFIG_SMP
#include <linux/spinlock_api_smp.h>  // real implementation
#else
#include <linux/spinlock_api_up.h>   // no-op stubs
#endif
```

In `spinlock_api_up.h`:

```c
#define spin_lock(lock)    do { (void)(lock); } while(0)
#define spin_unlock(lock)  do { (void)(lock); } while(0)
#define spin_trylock(lock) 1
```

`(void)(lock)` evaluates the argument (suppressing unused-variable warnings) and generates zero assembly. On a uniprocessor build, `spin_lock(&my_lock)` compiles to a single instruction that the optimizer removes immediately. The calling code looks identical in both builds, and the spinlock overhead is zero on platforms that do not need it. This is **zero-overhead abstraction**.

## Feature Test Macros: Unlocking POSIX Extensions

glibc (the GNU C Library) gates POSIX and GNU extensions behind feature test macros. Without them, many useful functions are hidden:

```c
// Without feature test macros: getline, strtok_r may be unavailable
#include <stdio.h>  // getline not visible

// With _POSIX_C_SOURCE 200809L: enables POSIX.1-2008 extensions
#define _POSIX_C_SOURCE 200809L
#include <stdio.h>  // getline now visible

// With _GNU_SOURCE: enables POSIX + GNU extensions + everything
#define _GNU_SOURCE
#include <stdio.h>
```

Common macros:

| Macro | Enables |
|-------|---------|
| `_POSIX_C_SOURCE 200809L` | POSIX.1-2008 (getline, strtok_r, ...) |
| `_XOPEN_SOURCE 700` | X/Open extensions |
| `_GNU_SOURCE` | POSIX + GNU extensions (strdup, asprintf, ...) |
| `_DEFAULT_SOURCE` | glibc defaults (modern replacement for _BSD_SOURCE) |

These must be defined **before** the first `#include`.

## Compiler and Platform Detection

Portable code detects the toolchain and platform:

```c
// Compiler detection
#if defined(__GNUC__) && !defined(__clang__)
    // GCC-specific code
    #define LIKELY(x)   __builtin_expect(!!(x), 1)
    #define UNLIKELY(x) __builtin_expect(!!(x), 0)
#elif defined(__clang__)
    // Clang-specific code
    #define LIKELY(x)   __builtin_expect(!!(x), 1)
    #define UNLIKELY(x) __builtin_expect(!!(x), 0)
#elif defined(_MSC_VER)
    // MSVC: no __builtin_expect
    #define LIKELY(x)   (x)
    #define UNLIKELY(x) (x)
#endif

// Platform detection
#if defined(__linux__)
    #include <sys/epoll.h>
#elif defined(__APPLE__)
    #include <sys/event.h>   // kqueue
#elif defined(_WIN32)
    #include <winsock2.h>    // IOCP
#endif
```

`__builtin_expect(cond, expected)` hints to GCC/Clang's branch predictor: `LIKELY(err != NULL)` tells the CPU to predict "true" for that branch, placing the hot path before the cold path in the generated code.
